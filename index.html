Boolean Structure

The 3×4 Matrix

Every block has a uniform structure: 3 sub-blocks × 4 clusters = 12 clusters per block.

Level Count Always
Blocks 4-6 per role Pillars of competency for the role
Sub-blocks 3 per block Concepts, Methods, Tools
Clusters 4 per sub-block Broad, Established, Recent, Specific
Terms 8-20 per cluster Exhaustive expansion

Blocks = Pillars of Competency

Blocks represent the core things someone must know, use, and do to perform the role. A viable candidate has signal in all blocks — the variation is in depth, not presence.

Block Test

Is this a core competency required to perform the role? If yes → block. If it's context (client industries), a filter (seniority), or a nice-to-have → not a block.

Do NOT create blocks for:

  • Leadership/Management — Use LinkedIn seniority filters instead
  • Client verticals — If the JD mentions "Finance, Sales, Retail" as domains the company serves, these are context, not candidate skills

Sub-blocks (Proxy Categories)

Every block has exactly 3 sub-blocks:

Sub-block What it captures Test
Concepts Field vocabulary, domain descriptors "What field is this?" — appears in team/org descriptions
Methods Techniques, algorithms, processes "What do they do?" — appears as skills
Tools Libraries, frameworks, platforms, benchmarks "What do they use?" — has a GitHub repo or docs site
Benchmarks Are Always Tools

SWE-bench, HumanEval, WebArena, MMLU → Tools. Never Methods (they don't describe how to do something) or Concepts (they don't describe what field someone is in).

Tiebreaker: If something passes multiple tests, use: Tools > Methods > Concepts

Clusters (Precision Tiers)

Every sub-block has exactly 4 clusters:

Cluster Purpose CSS Class
Broad High recall within the domain (none)
Established Mainstream terms (2+ years) (none)
Recent Cutting-edge (last 12-18 months) cluster-recent → adds ⚡
Specific Implementation-level, niche terms (none)
Broad ≠ Generic

Broad clusters catch people adjacent to this role's domain, not tech in general. Test: Would this term, searched alone, return mostly relevant candidates? If a term matches 50k+ general tech workers (PyTorch, Docker, Spark, pytest), it's too generic — don't include it.

2

Cluster Expansion

Every cluster must contain 8-20 terms. Apply these 8 expansion dimensions:

Dimension What to Include
Lexical variants Hyphenation, spacing, abbreviations
Synonyms Different words for same concept
Academic terminology Research paper language
Industry terminology Job posting language
Older terms Previous-generation phrasing
Emerging terms Newest developments (for Recent clusters)
Domain-specific Target domain manifestations
Compound terms Concept + common modifiers

Worked Example

Expanding "RLHF" for Methods (Established):

Expansion Process 
Start: RLHF

+ Lexical variants: "RL-HF", "RL from HF"
+ Full form: "reinforcement learning from human feedback"
+ Synonyms: "human feedback training", "preference-based training"
+ Academic: "preference learning", "reward learning"
+ Industry: "alignment training", "safety training"
+ Older terms: "interactive machine learning"
+ Compound: "RLHF pipeline", "RLHF fine-tuning"

Result:
("RLHF" OR "reinforcement learning from human feedback" 
OR "human feedback training" OR "preference-based training" 
OR "preference learning" OR "reward learning" 
OR "alignment training" OR "RLHF pipeline")
Expansion Must Stay Domain-Specific

Every expanded term must be role-relevant. Do NOT include: Generic infrastructure (PyTorch, Docker, AWS), generic testing (pytest, CI/CD), generic data (Spark, Airflow, ETL), generic methods (A/B testing, statistical testing). These catch everyone.

Output Format

Section Order

Section Purpose Contents
0. Role Summary Context Signal cards (Core Function, Domain, Tools)
1. Archetypes Guidance 3-4 personas with recipes + WHY
2. Boolean Reference Library 4-6 blocks with copy buttons on clusters
3. Footer Attribution Template link, role, date

Archetypes

Each archetype includes:

  • Recipe: Which Block → Sub-block (Cluster) combinations define this persona
  • WHY: Explains the combinatorial signal
WHY Quality

The WHY must explain what the combination signals — not restate what each cluster contains. A good WHY explains what false positives get filtered out by each added cluster.

Archetypes are guidance, not executable searches. Copy buttons appear on Boolean Reference clusters only.

Generation Order

Critical: Generation order ≠ Output order.

  1. Generate Boolean Reference first — Create all blocks with uniform 3×4 structure
  2. Synthesize Archetypes second — Map personas to existing blocks, write recipes + WHY
  3. Assemble Output — Arrange in final order (Archetypes appear before Boolean Reference)

Platform Syntax Reference

Note

The template generates platform-agnostic clusters. You copy clusters and adapt syntax for your target platform.

Quick Reference
LinkedIn Recruiter
  • AND, OR, NOT (caps)
  • "phrase", ()
  • Limit: ~2,000 chars
GitHub Search
  • AND, OR, NOT (caps)
  • "phrase", (), /regex/
  • repo:, org:, language:
Google X-Ray
  • OR (caps), "phrase"
  • NOT = -prefix
  • site:, inurl:

Iteration Playbook

Zero Results

  1. Remove clusters one at a time (start with most specific)
  2. Check for syntax errors (missing quotes, wrong operators)
  3. Try broader cluster tier (Established → Broad)
  4. Try Google X-Ray instead of native LinkedIn

Too Many Results

  1. Add another cluster as AND clause
  2. Use more specific cluster tier (Broad → Established → Recent)
  3. Add NOT clauses for observed false positives
  4. Use LinkedIn seniority filters

Wrong Results

  1. Check which terms are catching false positives
  2. Add NOT clauses for those patterns
  3. Try a different archetype's recipe
  4. Combine clusters from different blocks

Final Check

Before using generated output, verify:

Structure

  • 4-6 blocks, each representing a pillar of competency
  • 3 sub-blocks per block (Concepts, Methods, Tools)
  • 4 clusters per sub-block (Broad, Established, Recent, Specific)
  • 8-20 terms per cluster

Blocks

  • Each block is a core competency for the role
  • No leadership/management blocks
  • No client vertical blocks

Archetypes

  • 3-4 archetypes, each with recipe + WHY
  • WHY explains combinatorial signal, not cluster contents
  • Each archetype is a distinct sourcing path
  • No generic personas ("Technical Lead", "Domain Expert")

Content

  • No seniority terms in boolean clusters
  • No emojis in titles
  • Copy buttons on Boolean Reference clusters only
  • Benchmarks in Tools only (not Methods or Concepts)
  • No generic terms in Broad clusters (PyTorch, Docker, Spark)
  • All terms are domain-specific (not general tech)
